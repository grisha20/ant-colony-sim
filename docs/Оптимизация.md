## Пункт 3. Не сохранять феромоны в обычный snapshot.json - **[Выполнено]**

Что исправить:

Обычный autosave не должен записывать феромоны в `snapshot.json`.

Зачем:

Феромоны - временное runtime-состояние. Их не обязательно хранить между перезапусками. На текущем сохранении они занимают примерно 747 KB из 1.6 MB.

Файлы:

- `server/src/state/snapshot.ts`

Что сделано:

`saveWorldSnapshot(world)` теперь пишет:

```ts
toSnapshot(world, false)
```

То есть сохранение идет без феромонов.

Проверка:

- **[Выполнено]** `npm.cmd run check`

Риск:

Низкий. После перезапуска колонии потеряют старые запаховые тропы, но сама симуляция, муравьи, матка, комнаты, еда, паук и геномы сохраняются.

## Пункт 4. Добавить WebSocket backpressure - **[Выполнено]**

Что исправить:

Если клиент медленно принимает данные и у WebSocket накопилась большая очередь, сервер не должен бесконечно добавлять туда новые snapshots.

Зачем:

Один слабый или зависший клиент не должен раздувать память сервера.

Файлы:

- `server/src/net/socket.ts`

Что сделано:

Если `client.bufferedAmount > 2 MB`, сервер пропускает текущий broadcast для этого клиента.

Проверка:

- **[Выполнено]** `npm.cmd run check`

Риск:

Низкий. Медленный клиент может пропустить кадр, но следующий свежий snapshot все равно придет.

Проверка:

Риск:

Низкий. Цифры HUD обновляются вместе с новыми server snapshots.

## Пункт 7. Убрать повторные filter/map в горячем server tick - **[Не начато]**

Что исправить:

В горячем пути много повторных проходов по массивам:

- `world.ants.filter(...)`
- `world.colonies.map(...)`
- `world.underground.brood.filter(...)`
- повторные подсчеты живых муравьев;
- повторные поиски колонии по `colonyId`.

Зачем:

На 160 муравьях это терпимо. На 1000+ муравьях и нескольких колониях лишние проходы начинают съедать tick budget.

Файлы:

- `server/src/sim/step.ts`
- `server/src/sim/enemy.ts`
- `server/src/sim/world.ts`
- `server/src/sim/ant/step.ts`
- `server/src/sim/ant/combat.ts`
- `server/src/sim/ant/colony-state.ts`

Как исправлять:

1. Расширить `tickCache`.
2. Один раз за tick считать:
   - живых муравьев;
   - поверхностных муравьев;
   - муравьев по колониям;
   - бойцов;
   - рабочих у входа.
3. Использовать готовые значения вместо повторных `.filter`.

Ожидаемый эффект:

Меньше лишних аллокаций и проходов по массивам.

Риск:

Низкий-средний. Можно ошибиться в кешах и получить неверное поведение ролей или боя. Делать маленькими шагами.

## Пункт 8. Перейти на squared distance в радиусных проверках - **[Не начато]**

Что исправить:

Во многих местах используется `Math.hypot`, хотя нужен только ответ: объект внутри радиуса или нет.

Зачем:

`Math.hypot` считает квадратный корень. Для тысяч проверок за tick дешевле сравнивать квадрат расстояния.

Файлы:

- `server/src/sim/ant/*`
- `server/src/sim/enemy.ts`
- `server/src/ai/spiderBrain.ts`
- `server/src/sim/world.ts`

Как исправлять:

Ввести helper:

```ts
distanceSq(a, b)
isWithinRadius(a, b, radius)
```

Где можно менять:

```ts
distance(pos, enemy.pos) <= radius
```

на:

```ts
distanceSq(pos, enemy.pos) <= radius * radius
```

Где нельзя менять бездумно:

- нормализация движения;
- плавная сила отталкивания;
- score, где нужна реальная дистанция;
- сортировки по настоящей дистанции, если значение выводится или используется как длина.

Риск:

Низкий-средний. Ошибка в одном радиусе может сломать бой, паука или вход в гнездо.

## Пункт 10. Разделить durable snapshot и network snapshot - **[Не начато]**

Что исправить:

Сейчас один `toSnapshot` используется и для сохранения, и для отправки клиенту.

Зачем:

Для сохранения нужны долгоживущие данные. Для клиента нужны видимые данные прямо сейчас. Это разные задачи.

Файлы:

- `server/src/sim/world.ts`
- `server/src/state/snapshot.ts`
- `server/src/net/socket.ts`
- `shared/types.ts`

Как должно быть:

Durable snapshot:

- колонии;
- матки;
- расплод;
- комнаты;
- еда;
- паук;
- геномы/ссылки на геномы;
- минимум runtime-мусора.

Network snapshot:

- видимые муравьи;
- видимая еда;
- нужный underground;
- HUD;
- феромоны по необходимости.

Риск:

Средний. Это первый шаг к нормальному онлайну, но можно сломать загрузку или клиент.

## Пункт 18. LOD для колоний без наблюдателя - **[Не начато]**

Что исправить:

Колонии, на которые никто не смотрит, не должны симулироваться на уровне каждого муравья каждый tick.

Зачем:

Тысячи колоний невозможны в детальной симуляции одновременно.

Как должно быть:

- Активная колония: детальная симуляция.
- Далекая/невидимая колония: агрегатная симуляция числами.
- Offline-прогресс: событийная модель.

Файлы:

- пока отдельный дизайн;
- будущие `WorldInstance`, `ArenaShard`, `ColonySummary`.

Риск:

Высокий. Можно получить две разные игры: детальную и агрегатную.

## Пункт 19. Sharding / Arena instances - **[Не начато]**

Что исправить:

Не хранить тысячи колоний в одном `world.colonies[]`.

Зачем:

Массовый онлайн должен быть набором маленьких авторитетных арен, а не одним гигантским tick-loop.

Как должно быть:

- `ArenaShard` содержит ограниченное число активных колоний.
- Колонии без активного игрока живут как summary/LOD.
- Несколько shard-ов можно запускать в worker threads или отдельных процессах.

Риск:

Высокий. Делать только после стабилизации одиночной арены и сетевого projection.

## Пункт 20. Redis/PostgreSQL - **[Отложено]**

Что исправить:

Перейти от JSON-файлов к серверному хранению.

Зачем:

Для массового онлайна нужны аккаунты, владение колониями, история, экономика, persistence.

Почему отложено:

Сейчас важнее стабилизировать симуляцию и формат состояния. Ранний переезд в БД усложнит разработку.

Риск:

Высокий.
