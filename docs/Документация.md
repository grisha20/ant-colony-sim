> точечное подземелье) — приоритет у этого блока.

- **Две колонии-арена** (`world.colonies[]`): два враждующих улья (тёмные + красные), каждый со
  своим гнездом, геномом, эволюцией, фитнесом; общий food-феромон, личный home-феромон у каждой.
  Муравьи разных колоний дерутся при контакте (`antVsAntDamage`), голодная колония идёт войной.
  Паук — общая угроза. HUD показывает обе (колонки A/B); в подземелье переключатель «Гнездо A/B».
  Карта поверхности увеличена (~340×210).
- **Жизненный цикл матки + наследование**: у матки `stress` (от долго лежащих под ней яиц), `age`,
  `hp`. Есть **принцессы** (запасные матки). При смерти матки принцесса наследует трон, геном
  эволюционирует через `recordAndEvolve` (отбор+мутация), колония продолжает жить; нет принцесс →
  Game Over. Поколение закрывается по смене/смерти матки (таймер `generationMaxTicks` отключён как
  основной путь). При высоком стрессе обычные яйца у матки могут экстренно дорасти в слабых рабочих
  (`strength < 1`), а яйца-принцессы пополняют банк принцесс и не раздувают стресс матки. HUD:
  «Стресс матки», «Возраст матки», «Принцессы».
- **Копаемый underground (грид)**: подземелье — грид клеток (`UndergroundTile[][]`:
  soil/tunnel/chamber/entrance) с комнатами (`rooms`, у каждой `capacity`/`used`), задачами копки
  (`digTasks`) и счётчиком выкопанного (`dirtMound`). Рабочие роют комнаты по потребности
  (ясли/склад) и расширяют их при переполнении; землю выносят не после каждой клетки, а пачкой
  через `dirtLoad`/`dirtCarryBatch`. Стартовый вход начинается с верхней линии подземелья, а layout
  гнезда строится от `colonyId`, поэтому у тёмных и красных разные входы, маточные камеры и форма
  ходов. Ясли больше не берутся из фиксированной точки: планировщик ищет свободное место рядом с
  маткой, оценивает короткий ход и отсутствие пересечений с уже выкопанными комнатами. Ясли
  становятся пригодными частично: вместимость равна числу уже выкопанных chamber-клеток комнаты
  `room-nursery-*`, но не выше лимита 20 на одну камеру; начатая камера докапывается до чёткой
  формы. Один склад держит до 100 еды; при нехватке места создаётся новая отдельная складовая, а не
  бесконечное расширение старой. Рендер — разрез (тёмная земля, светлые ходы/камеры). Движение
  муравьёв идёт по реально выкопанным tiles; старые nav-точки сохранены как удобные целевые имена.
  Старые «точечные» камеры (`queenChamber`/`storage`/...) пока сосуществуют с гридом.
- **Баланс паука (без читов)**: муравьи видят паука и еду только в локальном радиусе
  (`antSpiderSightRadius`/`antFoodSightRadius`) — глобальное «чутьё» убрано, фуражировка по
  феромонам. Паук — спринтер с усталостью (`spiderSprintTicks`/`spiderTiredSpeed`/`spiderRecovery`),
  храбрее (`spiderMobRetreatCount: 5`), под давлением может есть/охотиться в отчаянии (у логова или
  при критическом голоде) — вышел из «петли смерти». Логово паука у края карты.

## 1. Что это

Онлайн-симуляция муравьиной колонии в духе SimAnt + idle/тамагочи. Игрок почти не управляет
муравьями напрямую, а наблюдает за жизнью колонии: муравьи ищут еду, носят её домой, матка
кладёт яйца, расплод вырастает в новых рабочих, на поверхности бродит паук-угроза, колония может
вымереть от голода. Игрок может вмешиваться — подкидывать еду кликом по карте.

Текущая стадия — одиночное ядро (один игрок, одна колония, одна карта). Без БД, доната,
полноценного онлайна на нескольких игроков, врагов и нейросетей.

---

## 2. Технологический стек

- **Сервер:** Node.js + TypeScript. Авторитетная симуляция, фиксированный тик 10/сек.
- **Клиент:** Vite + TypeScript + PixiJS (Canvas/WebGL). Только рендер.
- **Связь:** WebSocket (библиотека `ws`), порт 8787.
- **Состояние:** JSON-файл `snapshot.json` (живой мир сохраняется и подгружается при старте).
- **Клиент-сервер:** один монорепо с npm workspaces (`server`, `client`, `shared`).

---

## 3. Архитектура

Сервер — единственный источник правды. Каждый тик он считает всю симуляцию и шлёт клиенту
снимок мира (`WorldSnapshot`) по WebSocket. Клиент ничего не считает — только рисует то, что
пришло, и сглаживает движение между снимками.

Мир состоит из двух слоёв:

- **Surface (поверхность):** большая сетка 240×150 (размер в `config.ts`), рендерится на весь экран с камерой; тут живут феромоны, еда, паук и поиск еды.
- **Underground (муравейник):** боковой срез с реальной выкопанной сеткой. На старте есть только маточная камера, маленький склад и ход к поверхности; ясли появляются не сразу, а как задача копки по потребности.
  `diggable underground` хранит клетки `soil/tunnel/chamber/entrance`, `rooms`, `digTasks` и `dirtMound`.
  Рабочие берут доступные клетки из задач копки, копают их рядом с уже выкопанной областью, могут помогать на одном фронте и несут землю к входу пачками (`dirtLoad` до `dirtCarryBatch`); движение под землёй ограничено выкопанными tiles.

Слои связаны одной точкой — входом в муравейник (`entrance`). Муравей, дойдя до входа на одном
слое, переносится на вход другого.

## 5. Как запускать

Самый простой способ — bat-файлы в корне (двойной клик):

- **Запуск.bat** — включить сервер. Откроется окно с логами, не закрывай его.
- **Стоп.bat** — выключить сервер.
- **Рестарт.bat** — перезапустить (после правок кода/настроек).
- **Сброс.bat** — начать мир с нуля (удаляет `snapshot.json`, обучение сохраняется).
- **Сброс обучения.bat** — обнулить ОБУЧЕНИЕ (удаляет `genome.json`, `spider_genome.json`, `snapshot.json`) — для чистого эволюционного эксперимента.
- **Открыть порты.bat** — разрешить порты 5173/8787 в firewall (для игры по сети), запускать от админа.
- **Скопировать в GitHub.bat** — копирует проект в папку git-репозитория (без node_modules) и ставит зависимости.

Через командную строку (нужен установленный Node.js):

```bash
npm install     # один раз
npm run dev     # запуск сервера + клиента
npm run reset   # удалить snapshot.json (чистый старт)
npm run check   # проверка типов TypeScript
```

Клиент: `http://localhost:5173`  WebSocket: `ws://localhost:8787`

## 6. Игра по сети (Radmin VPN)

Второй игрок может смотреть мир со своего ПК (только наблюдение).

1. Оба в одной сети Radmin VPN.
2. На хосте запущен сервер; в окне есть строка **Network:** с IP (вид `26.x.x.x`).
3. Один раз открыть порты: **Открыть порты.bat** правым кликом → «Запуск от администратора»
   (разрешает входящие TCP на 5173 и 8787).
4. Второй игрок открывает `http://<Radmin-IP-хоста>:5173`.

Клиент сам подключается к данным по тому же хосту (`ws://<host>:8787`).

## 7. Как устроена симуляция

### Муравей (state-machine)
Простой агент без «сознания». Состояния: `idle`, `toEntrance`, `search`, `carry`, `deposit`,
`return`, `carryBrood`, `feed`, `dig`, `carryDirt`, `dead`. Дополнительно в снапшоте есть текущая работа `job`: `forage`, `nurse`, `dig`, `carryDirt`, `idle`.
Рабочий выходит из гнезда, по феромонам ищет еду, ест на
месте, берёт единицу и несёт в склад, кормит расплод, при голоде возвращается поесть.
У рабочего есть `strength`: обычные муравьи рождаются с `1`, экстренно доращенные у матки — слабее
(`queenRearStrength`). Сила множит наносимый урон и максимум переносимой еды. У копателя есть
`dirtLoad` — сколько выкопанных клеток земли он уже накопил перед рейсом к входу.

**Физическое поведение на поверхности:**
- **Взаимное расталкивание (Separation Force):** Муравьи на поверхности мягко отталкивают друг друга в небольшом радиусе (1.8 ед.), что предотвращает их скопление в одной точке при сборе еды, движении к входу или атаке на паука (муравьи облепляют паука кольцом).
- **Отскок от краев карты:** При соприкосновении с краем карты вектор направления муравья `heading` зеркально отражается, не давая ему застревать у границ.

### Питание (жизнь колонии завязана на склад)
Голодный рабочий сначала ест **на поверхности у источника еды** (доходит до еды → энергия
восстанавливается) — пока поверхностная еда есть, склад не тратится зря. Если поверхностной еды
нет, рабочий возвращается домой и ест **из склада** (фолбэк).

Ключевой принцип: **пока на складе есть еда — колония живёт.** Рабочие не умирают, матка кладёт
яйца, расплод растят. Поверхностная еда, подкормка игрока и труп паука — лишь способы **пополнять
склад**. Когда склад опустевает — наступает реальный голод: рабочие гибнут, матка со временем
погибает, а колония роем идёт на паука (см. ниже).

### Феромоны
Две сетки `Float32Array` только на поверхности: «к еде» и «к дому». Каждый тик — испарение
(умножение на коэффициент <1) и лёгкая диффузия на соседние клетки. Муравьи читают градиент и
пишут след, путей не хранят.

Для предотвращения автоколебаний («дерганья» муравьев на границах пикселей) выборка градиента сетки феромонов выполняется с помощью **билинейной интерполяции** (метод `getInterpolated`), что обеспечивает плавное изменение возвращающей силы, которая плавно сходит к нулю на вершинах феромоновых холмов. Также реализована механика **локального поиска**: если муравей стоит на сильном феромоне, но не видит еду напрямую, его случайное блуждание временно увеличивается, помогая ему сходить со следа в тупиках. При отсутствии еды на всей карте муравьи полностью игнорируют феромоны еды и переходят в режим чистой разведки.

### Подземелье: выкопанная сетка + комнаты
Подземелье теперь считается как боковой срез с grid-слоем: `soil`, `tunnel`, `chamber`, `entrance`.
На чистом старте выкопаны **маточная камера**, маленький **склад** и ход к поверхности. **Ясли**
не существуют сразу: когда у матки появляются яйца, колония создаёт `digTask` на маленькую ясли-камеру.
Рабочие копают только клетки, соседние с уже выкопанной областью, могут помогать на одном фронте и
копят несколько выкопанных клеток в `dirtLoad`, после чего несут пачку земли к входу (`dirtMound`).
Новые комнаты планируются не как жёсткие координаты, а как роевое решение: рядом с маткой/существующим
гнездом, с коротким органичным ходом и без наложения на другие комнаты. Ясли регистрируются как
комната уже после первой выкопанной chamber-клетки, но начатая камера докапывается до чёткой формы.
Одна ясли-камера вмещает до 20 яиц/личинок, один склад — до 100 еды; если места не хватает, создаётся
новая отдельная камера того же типа (`room-nursery-2`, `room-storage-2`...), а не одно большое общее
пространство. Подземное движение идёт по реально выкопанной сетке, а старые узлы (`entrance`,
`queenChamber`, `storage`, `nursery`) используются как целевые точки.

### Расплод (полная цепочка)
1. Матка кладёт яйцо под себя (камера матки).
2. Если яслей ещё нет, наличие яиц у матки создаёт задачу копки яслей.
3. После появления свободной выкопанной клетки яслей свободные няньки возвращаются вниз и несут яйца из маточной камеры в ясли.
4. В яслях яйцо инкубируется → личинка.
5. Личинку кормят в яслях (расход еды со склада) → вырастает во взрослого рабочего.
6. Новый рабочий уходит на поверхность добывать еду.
Защита от дедлока: перенос яиц в частично выкопанные ясли может вызвать возврат нянек с поверхности;
кормление личинок требует запас еды, а при голоде рабочие уходят добывать еду. Если стресс матки
превышает геномный порог `queenRearThreshold`, обычные яйца у матки могут дорасти в слабых рабочих
(`strength = queenRearStrength`), а яйца-принцессы — в принцесс в банке наследования.

### Распределение работы (роли)
Контроллер распределяет рабочих по ситуации (директивы `nurseTarget` / `activeTarget` / `diggerTarget`):

- **Есть еда на поверхности** → работают почти все (фуражируют). Нянек — до `maxNurses` (≈2) и только
  при наличии расплода; если расплода нет, эта пара остаётся у матки как стража.
- **Есть яйца у матки, но яслей нет** → появляется задача копки (`dig`/`carryDirt`), рабочие выкапывают
  маленькую ясли-камеру и выносят грунт к входу пачками. Если есть задача копки, контроллер держит
  минимум двух копателей даже при слабом гене `digFraction`, чтобы обе колонии не зависали на старте.
- **В яслях есть свободная выкопанная клетка, яйца ещё у матки** → часть рабочих возвращается с
  поверхности как `nurse`, чтобы перенести яйца в ясли.
- **Еды на поверхности нет** → часть рабочих (`scoutFraction`, ≈половина) уходит в разведку искать
  еду/паука, остальные **отдыхают в казармах** и стоят там (не наматывают круги).
- Лишние рабочие с поверхности возвращаются домой отдыхать, а не ищут бесконечно.

### Матка
Пока жива — кладёт яйца (при запасе еды и пока население < `nestCapacity`). Ест со склада; при
длительном голоде (`queenStarveBuffer`) погибает — кладка прекращается, колония постепенно вымирает.
Стресс матки зависит от переполнения яйцами в маточной камере: комфортный лимит задаёт
`queenEggComfortLimit`. Яйца-принцессы в этот счётчик не входят. Пока нет ни одной выкопанной клетки
яслей, переполнение не повышает стресс; дальше стресс может включить экстренное доращивание у матки.
Порог берётся из директивы `queenRearThreshold`: обычные яйца дают слабых рабочих, яйца-принцессы
дают принцесс в банк наследования.

### Еда
Несколько источников на карте с конечным запасом. **Автоспауна пока нет**, но запас пополняется
двумя способами: подкормкой игрока и трупом убитого паука. Источники с нулевым запасом считаются
пустыми.

### Подкормка игроком
В режиме «Поверхность» клик по карте отправляет на сервер команду `dropFood` (через тот же
WebSocket). Сервер валидирует координаты и создаёт новый источник еды в этой точке
(`addFoodSource`). Так игрок спасает голодающую колонию и влияет на среду. Число источников
ограничено сверху.

### Враг: паук (умный + эволюционирующий)
На поверхности живёт паук — отдельный хищный ИИ, симметричный колонии.

**Мозг** (utility-AI, `ai/spiderBrain.ts`): паук выбирает режим —
`feed / store / ambush / stalk / chase / retreat / wander` — по ситуации (голод, hp, плотность
муравьёв, дистанция до добычи/падали, наличие запаса) и держит решение, чтобы не дёргаться.
Движение **осмысленное** (к понятным целям), без случайного «DVD»-дрожания и без искусственных стенок
(только край карты, куда паук жёстко клампится — за карту уйти не может). Отступает только при реальной
угрозе: мобинг (несколько `fight`-муравьёв) или низкое hp; осторожность к толпе задаёт геном, а не
жёсткое правило.

**Своя пищевая цепочка (как у муравьёв): сперва еда, потом охота.** Паук — хищник и питается
**падалью** (`surface.carrion` — останки насекомых, разбросаны по карте и периодически респавнятся;
муравьи их НЕ едят) и **убийством муравьёв**. У паука есть **логово** (в кормовой зоне колонии) и
**запас** (`hoard`): сытый паук таскает падаль в логово — его рейсы пересекают территорию муравьёв,
отсюда стычки. Голодный сперва ест из запаса/падали, и только если еды нет — охотится на муравьёв.
Голод растёт со временем; **при долгой бескормице паук гибнет от голода**.

**Геном паука + эволюция** (`ai/spiderGenome.ts`): свой геном (агрессия, склонность к засаде,
дистанция погони, осторожность к рою, тяга к входу, терпение) с теми же приёмами, что у колонии —
архив топ-N, турнирный отбор, аддитивная мутация. Фитнес = убитые муравьи (+ выживание). Поколение
паука закрывается **по смерти (мобинг/голод) ИЛИ по таймеру жизни** (`spiderLifeMaxTicks`) — новый
паук рождается «умнее». Хранится отдельно в `spider_genome.json`. HUD показывает «Поколение паука» и
монотонный счётчик «Поколений (паук)».

**Связь с колонией:** сытая колония обходит паука (`spiderAvoidRadius`); голодная (склад пуст) роем
идёт на него и убивает, теряя муравьёв. Убитый паук → крупный труп-еда для муравьёв → позже новый
паук. Получается ко-эволюция «хищник ↔ рой».

### Колониальный ИИ (геном + эволюция)
Фундамент адаптивного интеллекта колонии — без нейросетей:

- **Геном** (`ai/genome.ts`) — набор «генов» (чисел) с жёсткими границами: доля нянек, доля копателей
  (`digFraction`), ранность экстренного доращивания у матки (`queenRearBias`), разброс поиска, порог
  атаки на паука, резерв кладки, порог голода, радиус обхода паука.
- **Адаптивный контроллер** (`ai/controller.ts`) каждый тик превращает геном в «директивы колонии»,
  которые `ant.ts`/`brood.ts` читают вместо части фиксированных констант. `diggerTarget` задаёт
  рабочее число копателей, а `queenRearThreshold` — стрессовый порог слабых рабочих/принцесс.
  Плюс копит **фитнес**
  колонии. Формула без «потолка»: добытая еда (главный навык) + убитые пауки (большой вес) + средняя
  популяция (умеренно) + выживание. Раньше доминировал «интеграл населения», но он упирался в потолок
  (вместимость×длительность) и замораживал эволюцию — теперь его вклад убран.
- **Эволюция (по таймеру ИЛИ смерти).** Поколение закрывается, когда умирает матка ИЛИ колония
  прожила `generationMaxTicks` — поэтому даже бессмертная процветающая колония продолжает
  эволюционировать, не дожидаясь гибели. Смерть матки → немедленный авто-рестарт мира. При закрытии
  поколения фитнес идёт в **архив топ-N**, новый геном = **аддитивная мутация** турнирно выбранного
  родителя, поколение+1, колония перезапускается фрешем.
- **Сохранение:** геном лежит в отдельном `genome.json`, переживает перезапуск сервера и `reset`
  (обнулить обучение — `Сброс обучения.bat`). HUD показывает «Поколение» (глубина родословной),
  **«Поколений (колония)»** — монотонный счётчик реально пройденных поколений, и «Лучш. фитнес».
  Симуляцию можно оставить на ночь — поколения копятся сами (если ПК не уходит в сон).

Это минимальная, но настоящая эволюция — фундамент «колония обучается и выживает». ML/RL отложено
(см. планы).

## 8. Сетевая оптимизация

Для плавности у удалённого клиента (по VPN):

- **Прорежение феромонов:** тяжёлая феромонная сетка (~20 000 чисел) шлётся не каждый тик, а раз в
  `pheromoneSendEveryTicks` тиков. Муравьи и HUD идут каждый тик; клиент переиспользует последние
  феромоны.
- **Интерполяция на клиенте:** муравьи плавно «доезжают» до новой позиции между снапшотами
  (сглаживание позиции и угла), рендер до 30 fps. При смене слоя (выход из норки / заход) позиция
  ставится сразу, без сглаживания — иначе спрайт «ехал» бы через всю карту.

Рендер оптимизирован: текстуры спрайтов кэшируются один раз, статичные слои строятся один раз,
муравьи/еда/расплод переиспользуются через пулы спрайтов. 

Спрайты муравьев (`ant` и `antCarry`) имеют размер 12x9 пикселей и абсолютно симметричны по вертикали, с четким разделением усов и лапок.
Для колоний используются **независимые пиксельные палитры** (темно-бурая для `colony-1` и сочно-красная для `colony-2`), генерируемые на лету. Это позволяет избежать использования WebGL-тинта, который делал цвета муравьев грязными и искажал цвет переносимой еды (еда остается ярко-желтой у всех колоний). Матка красной колонии также имеет свою красно-бурую текстуру.

Клиент **полноэкранный**, с **камерой**
(режимы «Слежение» за гнездом / «Свободно» — пан мышью + зум колесом / «К гнезду») и **отсечением по
видимой области** (рисуются только объекты и клетки в кадре) — поэтому большой мир тянет плавно.

## 9. Сохранение

Сервер раз в `snapshotSaveEveryTicks` тиков пишет весь мир в `snapshot.json`. При старте мир
подгружается из этого файла. Чтобы начать заново — **Сброс.bat** или `npm run reset`.

Геномы хранятся **отдельно** от мира: колония — в `genome.json` (текущий + архив топ-N + поколение),
паук — в `spider_genome.json`. Оба переживают перезапуск сервера и `npm run reset` — обучение не
теряется. Стереть обучение можно вручную или кнопкой **Сброс обучения.bat** (удаляет `genome.json`,
`spider_genome.json`, `snapshot.json`).

> Важно: при смене шкалы фитнеса старые геном-файлы становятся несовместимы (старые огромные значения
> не побить новыми) — после правки формулы фитнеса обучение надо обнулять (`Сброс обучения.bat`).

## 10. Баланс (config.ts)

Все числа вынесены в `server/src/config.ts`. Ключевые группы:

- **Сеть/тик:** `tickMs`, `pheromoneSendEveryTicks`, `snapshotSaveEveryTicks`.
- **Старт:** `startingWorkers`, `startingEggs`, `startingFoodStorage`, `nestCapacity`.
- **Движение/энергия:** скорости на поверхности/под землёй, `maxEnergy`, `energyDrainPerTick`,
  `refuelEnergyThreshold`.
- **Феромоны:** испарение, диффузия, сила и радиус запаха еды, вес градиента, разброс поиска.
- **Матка/расплод:** стоимость яйца, резерв еды для кладки, кулдаун кладки, время инкубации,
  скорость и стоимость кормления личинки, параметры голода/стресса матки, `queenRearStrength`,
  `queenRearTicks` и базовый `queenRearStressThreshold` (сейчас рабочий порог приходит из генома).
- **Подземелье/копка:** размеры grid/комнат, ширина тоннелей, `digProgressPerTick`,
  `digProgressPerTile`, `dirtCarryBatch`, вместимость одной ясли-камеры/склада, `maxDiggers` как
  абсолютный потолок-страховка.
- **Питание/склад:** `workerMealCost` (расход склада на кормёжку рабочего), пороги голода колонии
  (`starveFoodThreshold`, `starveStorageThreshold`).
- **Кормление игроком:** `playerFoodAmount`, лимит числа источников еды.
- **Паук:** скорости (блуждание/`spiderChaseSpeed`), hp, радиус/урон атаки, голод и агрессия,
  труп-еда, респаун; **падаль** (`carrion*`), **смерть от голода** (`spiderStarve*`), **логово и
  запас** (`spiderLair*`, `spiderHoard*`, `spiderCarry*`).
- **Распределение работы:** `maxNurses`, `scoutFraction`, `minForagers`/`maxForagers`,
  `spiderNearNestRadius`/`spiderNearNestPenalty`, радиус и лимит защиты роем (`defenseRadius`,
  `defenseMaxHelpers`, `antMobCount`).
- **Колониальный ИИ:** границы генов (`genomeGeneBounds`, включая `nurseFraction`, `digFraction`,
  `queenRearBias`), `colonyDeathMinTicks`, `generationMaxTicks`
  (длина поколения по таймеру), веса фитнеса (`fitnessFoodWeight`/`fitnessSpiderWeight`/
  `fitnessAvgPopWeight`/`fitnessSurviveWeight`), размер архива и турнира, скорость мутации.
- **ИИ паука:** границы генов паука (`spiderGeneBounds`), `spiderLifeMaxTicks` (поколение по таймеру),
  веса фитнеса (убийства/выживание), размер архива/турнира, скорость мутации.

## 11. Известные ограничения

- **Источники еды не возрождаются сами.** Запас пополняют только игрок (подкормка) и труп паука.
  Если игрок не кормит и склад опустевает — колония гибнет от голода. Автоспаун еды — кандидат на
  будущее.
- **Матка гибнет, когда опустевает склад** (долгий голод). Восстановления колонии после смерти
  матки нет.
- **ИИ обучается, но требует тюнинга.** Геномы и эволюция есть и у колонии, и у паука (ко-эволюция
  «хищник ↔ рой»); поколения теперь закрываются по таймеру ИЛИ смерти, так что прогресс идёт и без
  гибели. Баланс хищник↔рой и веса фитнеса — предмет дальнейшего тюнинга. Полноценный ML/RL — отложено.
- Нет подземного рытья по необходимости, ролей (солдаты/разведчики), онлайна на нескольких
  игроков, доната — отложено (см. планы).
